## 1. 核心理念与原则
- KISS：保持方案简单直接，避免过度工程化。
- 第一性原理：先分析问题本质，再选择最小可行实现。
- 事实优先：结论必须来自代码、配置与仓库事实。
- 渐进式开发：先构思方案，再审核，再拆任务，再执行。
- 最小范围改动：只改必要文件、必要代码段。

## 2. 推荐执行流程（Agent）
1. 调研相关代码与配置，列出事实。
2. 输出中文方案并提请用户审核。
3. 审核通过后，拆分任务并执行。
4. 执行中若出现不确定性，立即请求用户决策。
5. 执行后进行内部审查（准确性、完整性、最佳实践一致性）。
6. 汇报改动文件、关键决策与未决事项。

### 2.1 单任务技术执行流程（对应 2 第3步）

1. 按照分配任务编码实现
2. 核对编码实现与设计（流程图，技术实现设计）的匹配度
3. 回归测试（编译运行）
4. 代码审查（自审 + 与团队成员）
5. 落盘整体实现与测试验证文档
6. 拉取代码（同步其他成员的变更）
7. 代码提交前检查（git status 确认只包含必要文件）
8. **半自动化原则**：除非人类明确要求全自动化，否则只执行必要操作并留给人确认
9. 按照 git workflow 规范提交代码（中文注释说明）
10. 推送到远程仓库（git push）

## 3. 冲突优先级
- 用户最新明确指令 > 本文件 > 仓库习惯。
- 涉及风险操作（破坏性命令、安全或生产影响）必须二次确认。

## 4. 安全规范
- 禁止提交密钥、凭证、secrets 到仓库
- 生产环境操作必须二次确认
- 敏感信息处理：禁止在代码中硬编码密钥、密码、token

## 5. 任务交接规范
- 任务未完成需交接时，必须输出：已完成内容、待办事项、关键上下文
- 交接记录格式：使用 Markdown 表格记录进度
